# Dashboard Help

The Dashboard provides a concise, read-only overview of system state: key metrics, recent activity, upcoming requisitions, and critical alerts. This page describes exactly what each Dashboard panel shows and how to use the information to take action in the application. The Dashboard loads and refreshes when the page is opened or when you switch back to it.

## **Panels on the Dashboard (what's actually shown)**

The current Dashboard is composed of four primary panels (your deployment may show a subset):

- **Key Metrics (3×3 grid):** Compact cards with the following metric keys: `Total Items`, `Total Stock`, `Recent Adds`, `Low Stock`, `Expiring Soon`, `Ongoing Reqs`, `Requested Reqs`, `Active Reqs`, `Overdue Reqs`.
  - *Important definitions:* `Total Stock` is computed from received batches minus consumed/disposed movements plus returns, excluding disposed batches. `Low Stock` (metrics card) is an absolute count of items with current stock < 10 and > 0. `Expiring Soon` uses shared status logic (`EXPIRING`) so it matches the alerts panel windows (typically 180 days for consumables and 90 days for non-consumables). `Recent Adds` counts items modified in the last 7 days. Requisition counts reflect statuses in the database.

### Metric definitions (exact, technical)

- `Total Items`: Count of distinct Items in the `Items` table.
- `Total Stock`: Calculated as SUM(received quantities of non-disposed batches) - SUM(consumed) - SUM(disposed) + SUM(returned).
- `Recent Adds`: Items with `last_modified` in the last 7 days.
- `Low Stock` (metric): Items with computed current stock < 10 and > 0 (absolute threshold).
- `Expiring Soon`: Items currently in `EXPIRING` status from shared status logic.
- `Ongoing Reqs`: Requisitions with status IN (`requested`, `active`, `overdue`).
- `Requested / Active / Overdue Reqs`: Counts of requisitions with the named status.

### Display & retention limits (what you will actually see)

- **Activity:** The Dashboard shows a compact recent window for readability, while activity records are retained.
- **Schedule Chart:** Upstream query fetches up to 7 upcoming requisitions but the widget displays up to 5 rows for compactness.
- **Alerts tabs:**
  - `Summary`: shows up to 50 rows; tooltip notes additional rows when cap is reached.
  - `All Alerts`: full lifecycle alert list with category, due date, and days-left columns.

- **Recent Activity:** Two small tables: a single-row "Latest Activity" and an "Activity History" list. The activity feed shows `Description`, `User`, and `Time`. The Dashboard shows the latest entries from a full activity history.

- **Schedule Chart:** A compact table of upcoming requisitions. It lists status, requester, expected request, and expected return. Only requisitions with `expected_request >= now` and status in (`requested`, `active`, `overdue`) are returned; the widget displays a small number of upcoming rows (display limited for compactness).

- **Critical Alerts:** Read-only alert tables. `Summary` includes expiration/disposal/calibration alerts plus low stock. `All Alerts` focuses on lifecycle alerts only (expiration/disposal/calibration). Alerts are deduplicated and sorted so Critical items appear first; rows are color-coded by status. **Note:** Items with 0 stock (fully consumed or disposed) are excluded from alerts and alert tables.

## **How to read and interpret Dashboard content**

- The Dashboard is a summary, not an interactive reporting panel. Most elements are non-selectable and do not provide direct drill-down links.
- Use the `Inventory` page for per-item details (locations, batches, reservations) and `Reports` for exports and deep analysis.
- Tooltips and short text fields provide extra context (e.g., full activity descriptions appear on hover).

## **Alert & Low-stock logic (precise, factual details)**

- **Alert sources:** Lifecycle alerts are generated by the alert engine with labels such as `expiring`, `expired`, `disposal warning`, `disposal overdue`, `calibration warning`, and `calibration overdue`. Severity follows: within 7 days -> `Critical`, within 30 days -> `Warning`, else `Info`; already-due items are `Critical`. **Items with 0 current stock are excluded from alert generation.**
- **Low-stock (reports) vs Low-stock (metrics):**
  - The **Alerts** panel uses the low-stock report (`get_low_stock_data`) which applies percentage-based thresholds: for consumables items the threshold is 20% of original stock, for others 10% of original stock (items with no original stock are skipped). A numeric threshold may be used in reports, but the default behavior is percentage-based. **Items with 0 stock are excluded from low-stock alerts.**
  - The **Low Stock** metric card uses a simple absolute rule (current stock < 10 and > 0) to produce a single count.

## **Interacting and common workflows**

- **Identify a potential shortage:** Check `Key Metrics` and `Critical Alerts`. For details, open `Inventory` or run the `Low Stock` report in `Reports` to see per-item quantities and supplier info.
- **Handle an expiring/calibration alert:** Use the Alerts list to see which items are affected, then open the item in `Inventory` (or the calibration/expiration report) to review batches and plan action.
- **Investigate activity:** Copy the description/time, and open the related record (item or requisition) in its page to see the full audit trail.

## **Limitations & notes**

- Dashboard widgets are intentionally compact and read-only — the app places actions on dedicated pages where you can edit, export, or perform transactions.
- The Dashboard does not have an export button or per-row click-to-open actions — use `Reports` for exports and `Inventory`/`Requisitions` to act on records.
